# 法务群自动化需求验收差距报告

> 验收日期：2026-07-15
> 对照文档：《需求梳理(1).md》《技术方案.md》及后续金山文档业务要求
> 验收范围：当前代码、自动化测试与线上运行配置

## 1. 结论

当前版本已经打通“企业微信会话存档读取 -> 附件下载 -> 腾讯 OCR -> 业务事件/同步日志”的真实入站链路，管理端、鉴权、定时任务和 mock 业务演示也已具备。

但系统尚未形成可投入真实业务的完整闭环。最关键的缺口是：企业微信主动发送仍为 mock、真实群与案件/成员没有绑定、金山文档仍为 mock、结构化抽取仍是正则规则而非 OCR + LLM。原始 8 项核心需求中，没有一项达到真实生产环境下的完整验收状态；其中 7 项部分完成，商家提问超时提醒未实现。

## 2. 原始 8 项需求验收

| # | 需求 | 状态 | 已有能力 | 主要缺口 |
|---|---|---|---|---|
| 1 | 还款提醒 | 部分完成 | 案件到期扫描、提前提醒任务、目标用户字段 | 只有单一提前天数；缺还款日当天及不限档位规则、每档话术；线上发送为 mock |
| 2 | 自定义提醒 | 部分完成 | 已有创建 API 和到期发送任务 | 管理端没有创建入口；线上发送为 mock；未完成真实群验收 |
| 3 | 违约升级提醒 | 部分完成 | 逾期状态扫描、逾期 3 天升级任务 | 只有单一档位；缺可配置多档位和话术；线上发送为 mock |
| 4 | 违约状态同步 | 部分完成 | 案件状态变更和同步日志已实现 | 金山文档仍为 mock，未写入真实表格；没有文档侧回读对齐 |
| 5 | 已还金额同步 | 部分完成 | 可识别付款金额并累加已还金额 | 必须先匹配已有案件；真实样例未匹配案号时不会更新；金山文档仍为 mock |
| 6 | 资料抓取归档 | 部分完成 | 会话存档、媒体下载、腾讯 OCR 已真实跑通；文书业务路由已实现 | 结构化抽取仅正则；真实金山上传未接通；低置信度复核不完整 |
| 7 | 缴费持续跟踪 | 部分完成 | 缴费通知可生成 7 天每日提醒，独立缴费同步日志已实现 | 固定每日一次，缺可配置档位/话术；案件未匹配时不创建；缺明确的缴费完成与取消后续提醒流程；线上发送为 mock |
| 8 | 商家提问超时提醒 | 未实现 | 无 | 缺提问监控模型、我方回复判定、超时任务、管理员配置和 @ 提醒 |

需求依据：原始需求要求所有提醒支持可配置次数、时点和每档话术，并要求提问超过 5 分钟后提醒，见《需求梳理(1).md》第 16-27 行。当前生命周期实现只有单一提前天数和单一逾期升级天数，见 `app/services/case_lifecycle_service.py`；调度器仅有到期提醒、归档拉取和案件状态扫描，见 `app/core/scheduler.py`。

## 3. 后续金山文档需求验收

| 需求 | 状态 | 说明 |
|---|---|---|
| 判决书/调解书/裁定书重命名 | mock 已实现 | 已生成 `原告-被告{文书类型}.扩展名`；尚未处理真实文件重名冲突 |
| 上传到 `致和法务/判决书文件` | mock 已实现 | 已生成上传 payload 和同步日志；线上 `KDOCS_MODE=mock`，没有真实文件 |
| 开庭传票写入开庭时间表 | mock 已实现 | 已抽取开庭时间并生成 `sort_by=开庭时间` payload；没有真实查询、插入和重排验收 |
| 文书写入强制执行进度表 | mock 已实现 | 已生成案号、原告、被告、类型、链接、摘要等 payload；没有真实写入 |
| 缴费通知/截图独立登记 | mock 已实现 | 已生成独立缴费登记 payload；真实样例缺案号时无法关联案件 |
| 失败记录与重试 | 已实现 | 同步日志保存失败状态并支持重试；幂等键只有普通索引，仍可能重复写入 |

金山适配器当前通过自定义网关 `/kdocs/{operation}` 调用，并非直接对接金山官方接口。切换真实模式前需要明确是部署该网关，还是将主服务改为直接调用金山开放平台。

## 4. 已完成的基础能力

- 企业微信会话内容存档官方 SDK 已接入真实模式，支持断点拉取、消息解密和媒体下载。
- 腾讯 OCR sidecar 已接入真实模式，线上附件能够完成 OCR。
- 归档群白名单、启用/停用、特殊消息识别群名已实现。
- 案件、消息、事件、媒体、提醒、发送日志、同步日志等基础模型和管理端页面已具备。
- 案件状态流转、已还金额累加、同步失败记录和重试机制已具备。
- API Key、RBAC、健康检查、数据库迁移和定时任务已部署。
- mock 演示链路可覆盖文书、开庭、缴费通知和付款截图。

## 5. 上线阻塞项

### P0：真实业务无法闭环

1. **企业微信发送端未接通**
   `WECOM_SEND_MODE` 线上仍为 `mock`。代码只支持 `mock` 和固定群机器人 `webhook`，还没有技术方案中的 PC Hook/发送 sidecar。提醒即使到期，也不会真正发到目标外部群。

2. **案件没有绑定真实群和成员**
   当前启用的真实归档群与现有演示案件的 `group_id` 不一致，真实 OCR 记录也没有匹配到 `case_id`。需要提供案件与 `roomid`、债务人 userid、法务 userid 的首次人工绑定功能，否则无法精确发群和 @ 人。

3. **金山文档未真实写入**
   线上仍是 `KDOCS_MODE=mock`。需要金山访问凭证、空间 ID、四类目标文件夹/表格 ID、字段映射，并完成真实写入验收。

4. **OCR 后缺少 LLM 结构化抽取**
   当前结果标记为 `regex_v2`。真实文书中案号出现空格、版式变化时会漏识别，原告/被告也可能错位。需要接入 LLM 结构化输出、置信度和人工复核队列。

### P1：核心需求不完整

5. **提醒规则、档位和话术模板未建模**
   需要 `reminder_rule`、档位、模板、启停配置，以及管理端编辑页面；还款提醒、违约升级和缴费跟踪统一使用该规则引擎。

6. **群类型自动识别和功能路由未实现**
   当前群表只有启用状态和显示名，没有“商家群/被告群”、功能集、超时阈值和管理员。原需求中的群名关键词自动接入、白名单/黑名单例外尚未完成。

7. **商家提问超时监控未实现**
   需要记录商家提问、判断我方是否回复、5 分钟超时扫描，并 @ 指定管理员。

8. **管理端业务配置不完整**
   缺群类型、案件群成员绑定、提醒档位、话术模板、关键词、提问超时和管理员配置；自定义提醒虽有 API，也没有页面入口。

9. **缴费跟踪缺少完成闭环**
   需要独立跟踪状态、截止时间、缴费完成确认，并在确认后取消未发送的后续提醒。

### P2：交付稳健性

10. 文书同名时追加案号或消息 ID，避免覆盖。
11. 将同步幂等键设为唯一约束，并在重试/回放时防止金山重复记录。
12. 补齐金山表格回读或明确改为单向写入，避免“与本地库双向对齐”的验收歧义。
13. 增加发送端掉线、OCR/金山连续失败、归档拉取停滞的外部告警。

## 6. 建议交付顺序

1. 先做案件与真实群/成员绑定，同时接通真实发送 sidecar，完成一条真实提醒。
2. 接通金山真实环境，逐一验收文书上传、开庭表、强执表和缴费表。
3. 接入 LLM 抽取与人工复核，使用真实样例回归案号、当事人、金额和开庭时间。
4. 建立统一提醒规则/模板和管理页面，替换当前固定 7 天与单档位逻辑。
5. 完成群类型自动路由和商家提问超时提醒。
6. 做全链路验收、异常重试、告警、部署与回滚演练后形成正式交付包。

## 7. 下一阶段需要的外部信息

- 金山文档真实访问方式：自定义网关地址或允许直接接官方 API。
- 金山访问令牌、空间 ID、判决书文件夹 ID、开庭表 ID、强制执行表 ID、缴费登记表 ID及列字段映射。
- 企业微信真实发送方案确认：PC Hook sidecar，或可加入目标外部群的群机器人 webhook。
- 每个案件对应的真实群、债务人 userid、法务/管理员 userid。
- LLM 服务地址、模型、API Key，或确认复用现有内部模型网关。
- 三类提醒的默认档位和正式话术，以及商家提问超时的管理员名单。
