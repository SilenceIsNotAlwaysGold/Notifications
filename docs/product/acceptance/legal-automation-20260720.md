# 法务群自动化项目整体验收与收口报告

> 验收日期：2026-07-20
> 验收对象：当前代码、自动化测试、线上配置和线上业务数据
> 需求依据：原《需求梳理(1).md》《技术方案.md》的既有验收记录，以及后续企业微信、金山文档业务要求

## 1. 验收结论

当前项目已经完成可演示的 Mock 业务闭环，并且企业微信会话内容存档、媒体下载、腾讯 OCR 已进入真实模式。案件、消息、事件、附件、提醒、发送日志、金山同步日志和管理端等基础能力也已形成。

但当前版本仍不能作为“真实业务全自动闭环”正式交付。主要原因是：

1. 企业微信提醒发送仍为 Mock；官方机器人 sidecar 虽已连接，但向目标外部群发送尚未成功验收。
2. 本地已补案件编辑、真实群和成员绑定能力，但线上尚未执行绑定，OCR 结果仍未稳定驱动真实案件和提醒。
3. 金山文档仍为 Mock，同步日志中的 `success` 不代表文件或表格已经真实写入金山。
4. 本地已补 OpenAI 兼容的 LLM 结构化抽取、校验和正则回退，但尚未配置生产模型，完整人工复核流程仍缺失。
5. 原始需求中的可配置提醒档位、话术模板和商家提问 5 分钟超时提醒尚未完成。
6. 当前大量真实集成代码尚未提交，部署包也没有完整包含 Node 机器人 sidecar 和企业微信官方 SDK。

按里程碑判断：

| 里程碑 | 状态 | 结论 |
|---|---|---|
| M1 Mock 演示版 | 已完成 | 一键演示、案件、事件、媒体、提醒和同步日志链路具备 |
| M2 Mock 业务验收版 | 基本完成 | 四类核心材料和字段抽取已有测试，复杂真实样例仍需扩充 |
| M3 金山真实写入版 | 未完成 | 仍为 Mock，缺真实网关、凭证、目标 ID 和现场验收 |
| M4 企业微信真实读取版 | 入站已完成 | 归档、媒体和 OCR 已真实运行；真实外部群发送未闭环 |
| M5 可部署交付版 | 部分完成 | 鉴权、迁移、健康检查和调度已有；告警、备份、完整打包和交付提交未收口 |

## 3. 金山文档需求完成度

| 需求 | 当前状态 | 缺口 |
|---|---|---|
| 判决书/调解书/裁定书识别 | Mock 已实现 | 生产真实样例仅见判决书，调解书和裁定书尚未现场验收 |
| 重命名为 `原告-被告{文书类型}.扩展名` | 已实现 | 同名文件没有追加案号或消息 ID，真实上传可能覆盖或冲突 |
| 上传 `致和法务/判决书文件` | Mock 已实现 | 没有真实文件上传结果和可访问链接 |
| 开庭传票写入 `致和法务/开庭时间` | Mock 已实现 | 生产没有开庭传票数据；目前只发送排序提示，未在本服务查询和重排现有记录 |
| 文书写入强制执行进度表格 | Mock 已实现 | 没有真实金山写入及字段映射验收 |
| 缴费通知/截图独立登记 | Mock 已实现 | 真实表格未写入，案件匹配失败时无法形成完整跟踪闭环 |
| 失败可重试 | 已实现基础能力 | 幂等键不是唯一约束，回放或重试可能产生金山重复记录 |

`KDocsAdapter` 当前依赖自定义 `/kdocs/...` 网关协议，仓库中没有该网关实现。因此只有以下两条路可选其一：

- 部署并维护金山网关，由当前主服务调用网关。
- 改造主服务，直接对接金山开放平台的鉴权、文件和表格 API。

在该选择落定前，`KDOCS_MODE=real` 不能视为可交付能力。

## 4. 企业微信链路现状

### 已完成

- 企业微信会话内容存档官方 SDK 接入，支持断点拉取和解密。
- 媒体文件真实下载和腾讯 OCR 识别。
- 归档群发现、启用/停用和 `#群名识别群 群名` 指令。
- 官方 AI 机器人 WebSocket sidecar、健康检查和群白名单。
- 线上主服务、调度器和机器人 sidecar 已运行。

### 尚未完成

- 线上 `WECOM_SEND_MODE=mock`，提醒日志中的 `sent` 只是 Mock 成功。
- 官方机器人向外部群测试返回发送频率限制错误，尚无一条真实外部群成功发送记录。
- 当前两个启用的真实群没有绑定现有案件；线上案件仍使用演示群 ID 和演示成员 ID。
- 本地管理端已支持案件编辑、真实群和成员绑定，但尚未部署并用线上真实案件验收。
- 还没有确定并验收最终外部群发送方案：官方机器人、客户群群发任务，或专用发送账号 sidecar。

## 5. 线上数据反映的问题

本次检查到的线上数据包括 29 条群消息、46 个业务事件、17 个媒体文件、7 个提醒和 62 条文档同步日志。17 个媒体均完成 OCR，说明真实入站技术链路已经工作。

但这些数据也暴露了四个问题：

1. 46 个事件中有 27 个为 `unknown`，约占 59%，正则分类和字段抽取对真实材料的覆盖不足。
2. 7 个提醒均为缴费跟踪并标记为已发送，但发送模式为 Mock，业务人员实际上没有收到提醒。
3. 61 条金山同步日志标记成功，但金山模式为 Mock，不能当作真实写入凭证。
4. 真实启用群与案件绑定数量为 0，导致真实消息无法稳定更新案件、生成提醒或准确 @ 人。

另外，生产环境尚未出现开庭传票、调解书和裁定书的完整样例，不能仅凭单元测试认定这些分支已经通过现场验收。

## 6. 管理端缺失能力

- 在目标环境执行真实案件与群、债务人、法务人员绑定，并核对历史材料回填结果。
- 自定义提醒创建、编辑、取消。
- 还款、违约、缴费提醒的档位、次数、时点和话术模板配置。
- 群类型配置，例如商家群、被告群，以及每类群启用的业务功能。
- 商家提问超时阈值、负责人和升级对象配置。
- OCR 人工复核队列：查看原文、修正字段、确认分类、重新执行业务同步。
- 金山目标、字段映射、同步失败详情和人工重试管理。
- 明确区分 Mock 发送/同步与真实成功，避免管理端产生误判。

## 7. 交付工程风险

### P0：正式使用前必须完成

1. 绑定至少一个真实群、真实案件和真实成员，并跑通真实材料到案件的闭环。
2. 确定外部群发送方案，成功发送一条真实提醒并正确 @ 目标人员。
3. 接通金山真实环境，现场验收四个目标位置的文件和表格记录。
4. 将 Mock 的 `sent/success` 与真实成功分开标记，防止运营误判。
5. 提交当前真实集成代码，清理交付边界，保证服务器版本能够从 Git 重建。

### P1：核心需求补齐

6. 建立统一提醒规则和话术模板，并补管理端配置。
7. 实现商家提问超过 5 分钟未回复的监控和提醒。
8. 配置并验收 LLM 结构化抽取，继续实现可操作的人工复核流程。
9. 完成缴费跟踪状态机，缴费完成后停止剩余提醒。
10. 增加文书重名策略和同步幂等唯一约束。

### P2：可运维交付

11. 增加归档停滞、OCR/金山连续失败、机器人掉线和发送失败的外部告警。
12. 制定附件留存、删除、访问审计和本地存储加密策略。
13. 建立数据库与附件备份、恢复演练和回滚说明。
14. 将 API、Node 机器人 sidecar、企业微信 SDK 和迁移纳入同一部署清单或安装脚本。
15. 收紧依赖版本，跟踪 `uv.lock`，让发布检查使用固定运行环境。
16. 将详细配置健康信息限制给管理员，仅公开最小健康状态。

## 9. 建议收口顺序

1. **真实业务绑定**：部署案件编辑和群/成员绑定页面，导入并绑定一条真实案件。
2. **发送闭环**：确定方案并在目标外部群成功发送、@ 人、记录回执。
3. **金山闭环**：确定直连或网关，配置四类目标，逐条真实写入验收。
4. **识别质量**：配置 LLM 并补复核队列，用判决书、调解书、裁定书、传票、缴费材料做回归。
5. **原始需求补齐**：提醒规则/话术、缴费完成、商家提问超时。
6. **交付工程收口**：告警、备份、部署脚本、安全检查、提交和版本标记。

## 10. 正式交付验收门槛

满足以下条件后，才建议将版本标记为“可交付”：

- 至少一个真实外部群完成“消息归档 -> OCR -> 案件匹配 -> 业务动作 -> 提醒发送”的全链路。
- 判决书、调解书、裁定书、开庭传票、缴费通知和付款截图均有真实样例验收记录。
- 金山四个目标位置均有真实文件或记录，并验证失败重试不会重复写入。
- 还款、违约和缴费提醒规则可由管理端配置；缴费完成可停止后续提醒。
- 商家提问超时提醒可配置、可触发、可关闭。
- 管理端清楚展示 Mock、待复核、失败、真实成功四类状态。
- 全量测试、数据库迁移、配置检查、备份恢复和回滚演练通过。
- 服务器可从已提交代码和交付文档完整重建，不依赖未提交的现场文件。
