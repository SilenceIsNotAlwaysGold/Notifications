# 企业微信接入可行性说明

本文用于第十六阶段 POC：验证企业微信外部法务群是否可以被系统可靠接入。

## 1. 企业微信群机器人 Webhook

企业微信群机器人 webhook 只能作为发送消息出口。

- 可以用于向群内发送文本、markdown 等机器人消息。
- 不能读取群消息。
- 不能作为群消息监听方案。
- 对外部群是否可用，需要客户在目标外部群中实际验证。
- 如果目标外部群不能添加机器人，则 webhook 方案不可用于该群提醒。

因此，webhook POC 的验证目标是：目标外部群是否能添加机器人，以及系统通过 webhook 发送的测试消息是否能被群成员看到。

## 2. 企业微信会话内容存档

企业微信会话内容存档是读取群消息的官方路线。

真实接入前需要客户确认并提供：

- 企业 ID `corp_id`
- 会话内容存档 `secret`
- private key 或 private key 文件
- `public_key_ver`
- 企业微信后台已开通会话内容存档权限

还必须进一步确认：

- 会话内容存档是否覆盖外部群。
- 参与外部群聊的客户员工是否在存档范围内。
- 是否可以拉取图片、文件、PDF 等媒体。
- 会话内容存档 SDK / API 是否允许当前业务所需的消息类型。

如果客户没有会话内容存档权限，系统无法通过官方方式读取企业微信外部群消息。

## 3. 官方 CLI 发送

企业微信官方维护的 `WecomTeam/wecom-cli` 已提供 `msg send_message`，群聊目标支持 `wr...` 群 ID。本项目支持 `WECOM_SEND_MODE=wecom_cli`，并直接复用会话内容存档发现的官方群 ID。

该通道不要求手机常驻，但有明确的开放范围限制：官方当前说明中，10 人及以下企业可使用消息、文档、日程、会议和待办等能力；10 人以上企业默认只提供文档和待办 CLI 能力。因此必须在目标企业执行 `wecom-cli init`、`wecom-cli msg --help`，再向无敏感内容的测试群实发，确认消息品类确实获权。

启用后仍须满足：

- 群已在管理后台“归档群”中人工启用。
- 只发送文字，单条不超过 2048 字节。
- 使用单群/全局每日上限、串行限速、失败熔断和发送审计。
- CLI 加密配置目录只保存在服务器，不进入 Git 或管理端。

## 4. 官方智能机器人 WebSocket

企业微信团队维护的 OpenClaw 插件支持智能机器人 WebSocket，并通过 `aibot_send_msg` 主动发送到群聊。它是第二条官方 POC 路线，不需要手机常驻，但机器人必须拥有目标会话的可达权限；自建应用 HTTP 接口对 `wr...` / `wc...` 外部群常见 `86008`，不能把内部应用群发能力等同于外部群直发。

该路线适合在 `wecom-cli` 没有 `msg` 能力时使用。本项目已实现独立的 `wecom_bot_sidecar`，仅依赖官方 `@wecom/aibot-node-sdk`，不把 OpenClaw 引入主业务服务；主服务通过 `WECOM_SEND_MODE=wecom_bot` 调用。启用前必须完成 WebSocket `authenticated` 探针、目标群双重白名单和无敏感内容测试消息验收。

## 5. 官方客户群群发任务

`externalcontact/add_msg_template` 可以由服务端自动创建客户群群发任务，但调用接口不会直接把消息发进客户群，仍需对应成员在企业微信客户端确认发送。它最合规、可追踪，适合作为 CLI/机器人不可用时的生产兜底；系统自动生成任务和催办，员工只做一次确认。

## 6. 可选的专用通知账号客户端自动化

当业务明确要求“无人值守直接发送到指定外部群”时，官方客户群群发任务仍需要成员确认，不能满足完全自动直发。项目因此保留可选的 `wecomapi` 兼容发送模式，可连接自托管 Android RPA 网关，但只允许作为隔离的外发通道：

- 使用独立企业微信成员“致和法务通知助手”，不使用律师主账号。
- 会话读取继续使用官方会话内容存档。
- 每个官方归档 `roomid` 必须显式映射协议 `RoomID`，未映射时阻止发送。
- 只发送必要的文字提醒，不上传法务文件。
- 启用串行限速、每日上限、失败熔断和发送审计。
- 优先采购私有化部署；SaaS 模式不得承载真实敏感材料。
- 接受其非官方 UI 自动化带来的账号风控、客户端升级适配和可用性风险。

## 7. 不建议方案

不建议使用以下方案读取企业微信群消息：

- 个人号 hook
- 模拟网页版
- 抓 cookie
- 非官方客户端监听
- 无隔离账号、无限速、无审计的逆向协议或非官方注入方式

原因：

- 稳定性差，容易因企业微信升级失效。
- 合规风险高。
- 容易触发封控。
- 难以向客户安全、审计和法务部门解释。
- 不适合处理判决书、付款截图等敏感材料。

## 8. 当前系统状态

当前系统已经具备以下工程基础：

- 已有企业微信群机器人 webhook 发送 adapter。
- 已有官方 `wecom-cli` 文本发送 adapter，直接使用官方群 ID，并带白名单、限速、上限和熔断保护。
- 已有可选 `wecomapi` 兼容文本发送 adapter、自托管网关、发送目标映射、限速、每日上限和熔断保护。
- 已有企业微信会话内容存档 adapter，可通过 SDK sidecar 拉取已解密归档消息。
- 已内置企业微信官方 Linux x86 SDK v3.0 backend，支持消息解密和媒体分片下载。
- 已有 `/api/v1/legal/wecom-archive/replay` 回放接口，可模拟归档消息进入业务链路。
- 已有媒体文件、OCR、提醒、腾讯文档 mock 同步等后续处理链路。

但需要明确：

- 当前尚未验证真实企业微信外部群消息读取。
- 主服务不直接加载企业微信官方 SDK，真实拉取由仅监听本机的 SDK sidecar 承担。
- 当前尚未验证客户是否具备会话内容存档权限。
- 当前尚未验证目标外部群是否允许添加群机器人。
- replay 模拟成功不代表已经真实接入企业微信生产环境。

## 9. POC 建议顺序

1. 安装并初始化官方 `wecom-cli`，执行 `wecom-cli msg --help` 探测消息权限。
2. 若有权限，向已启用的测试外部群发送虚构提醒，确认 `wr...` 群 ID 可达。
3. 若无权限，尝试让官方智能机器人加入测试外部群，验证 WebSocket 主动发送。
4. 若机器人不可用，选择“客户群群发任务 + 成员确认”作为官方兜底。
5. 只有业务必须完全无人值守且接受风险时，才启用隔离通知账号和 Android RPA。
6. 所有路线的读取侧继续使用官方会话内容存档，不使用客户端自动化读取敏感材料。
7. 完成错群保护、限速、熔断、账号离线和停用开关测试后再灰度。
